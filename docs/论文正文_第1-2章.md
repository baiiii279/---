# 第1章 开发环境与技术

## 1.1 开发环境与工具

### 1.1.1 Visual Studio Code

Visual Studio Code（简称VS Code）是一款由微软开发的轻量级跨平台代码编辑器，支持Windows、macOS和Linux操作系统。VS Code拥有丰富的扩展生态系统，支持Python、JavaScript、TypeScript等多种编程语言的语法高亮、代码补全、调试等功能。本项目使用VS Code作为主要的代码编辑工具，配合Python扩展和ESLint等插件，提高了开发效率和代码质量。

### 1.1.2 Python 3.10

Python是一种解释型、面向对象的高级编程语言，以其简洁的语法和丰富的第三方库生态著称。本项目使用Python 3.10作为后端开发语言，利用其类型提示（Type Hints）特性增强代码可读性和可维护性。Python在人工智能和Web开发领域拥有广泛的应用，FastAPI、CrewAI等核心框架均基于Python构建。

### 1.1.3 Node.js 18

Node.js是一个基于Chrome V8引擎的JavaScript运行时环境，采用事件驱动、非阻塞I/O模型，非常适合构建高性能的Web应用。本项目使用Node.js 18作为前端开发环境，配合npm包管理器管理React及相关依赖。Vite开发服务器基于Node.js运行，提供了毫秒级的热模块替换（HMR）能力。

### 1.1.4 MySQL 8.0

MySQL是一种开源的关系型数据库管理系统，以其高性能、稳定性和易用性被广泛应用于Web应用开发。本系统使用MySQL 8.0作为数据存储方案，利用其InnoDB存储引擎支持事务和外键约束，保证数据的一致性和完整性。通过SQLAlchemy ORM技术与Python后端交互，简化了数据库操作。

## 1.2 相关技术

### 1.2.1 FastAPI

FastAPI是一个基于Python的现代Web框架，专为构建高性能API而设计。它基于Starlette（异步Web框架）和Pydantic（数据验证库）构建，具有以下特点：

- **高性能**：与Node.js和Go相当的性能，是Python最快的Web框架之一。
- **自动文档生成**：自动生成OpenAPI（Swagger）交互式API文档，便于调试和测试。
- **类型安全**：利用Python类型提示进行请求参数验证和响应模型校验，减少运行时错误。
- **异步支持**：原生支持异步请求处理，适合处理I/O密集型任务如数据库查询和API调用。

本项目使用FastAPI作为后端框架，提供RESTful API接口，并利用其异步特性实现SSE（Server-Sent Events）实时推送Agent执行状态。

### 1.2.2 React 18

React是由Meta（原Facebook）开发的用于构建用户界面的JavaScript库，采用组件化开发模式。React 18引入了并发渲染（Concurrent Rendering）机制，提升了用户界面的响应性能。本系统使用React构建前端单页应用（SPA），配合React Router实现页面路由管理，利用Hooks（如useState、useEffect）管理组件状态和副作用。

### 1.2.3 Vite

Vite是一个现代化的前端构建工具，利用浏览器原生ES模块导入能力实现极速冷启动。相比传统的Webpack打包工具，Vite在开发环境中无需整体打包，仅在浏览器请求时才编译所需模块，实现了毫秒级的热更新响应。本系统使用Vite作为前端构建工具，配合TypeScript和React插件提供完整的开发体验。

### 1.2.4 CrewAI

CrewAI是一个专门用于编排多智能体（Multi-Agent）协作的Python框架。它借鉴了敏捷开发中的"Scrum"理念，允许开发者定义具有特定角色、目标和能力的AI Agent，并通过Task和Crew将它们组织起来协同工作。CrewAI的主要特性包括：

- **角色驱动**：每个Agent拥有独立的角色（Role）、目标（Goal）和背景故事（Backstory），使LLM能更准确地执行特定任务。
- **任务编排**：支持顺序执行（Sequential）和层次化（Hierarchical）两种流程，满足不同复杂度的协作需求。
- **工具集成**：Agent可以调用外部工具（如搜索API、文件读写等），扩展LLM的能力边界。
- **LLM无关**：支持接入不同的大语言模型提供商，包括OpenAI、DeepSeek、Claude等。

本系统利用CrewAI编排5个Agent（文献解析Agent、大纲生成Agent、内容撰写Agent、润色Agent、引用检查Agent），实现端到端的论文写作自动化。

### 1.2.5 DeepSeek API

DeepSeek是由深度求索（DeepSeek）公司开发的大型语言模型。DeepSeek API提供了与OpenAI API兼容的接口，支持文本生成、对话等自然语言处理任务。其优势包括：

- **性价比高**：相比海外API提供商，DeepSeek的定价更具竞争力，适合教育场景下的开发实践。
- **中文能力强**：针对中文语境进行了专门优化，在学术写作场景下表现良好。
- **兼容性好**：兼容OpenAI API格式，便于与CrewAI等框架集成。

本系统使用DeepSeek作为Agent层的大语言模型驱动，通过CrewAI的LLM配置接口接入，为每个Agent提供自然语言理解和生成能力。

### 1.2.6 SQLAlchemy

SQLAlchemy是Python生态中最成熟的对象关系映射（ORM）框架之一。它将数据库表映射为Python类，将记录映射为对象实例，将查询语句映射为Python表达式。本系统使用SQLAlchemy 2.0的声明式映射风格，定义6张数据表模型，配合PyMySQL驱动与MySQL数据库交互。通过依赖注入（Dependency Injection）管理数据库会话生命周期，确保资源正确释放。

### 1.2.7 python-jose 与 passlib

python-jose是一个用于处理JSON Web Token（JWT）的Python库，支持HS256、RS256等多种签名算法。passlib是一个通用的密码哈希库，封装了bcrypt、argon2等哈希算法。本系统使用python-jose生成和验证JWT令牌实现用户认证，使用passlib的bcrypt算法对用户密码进行哈希存储，确保用户信息安全。

# 第2章 系统需求分析

## 2.1 功能需求分析

本系统——PaperCraft多智能体论文写作助手——是一个基于Multi-Agent架构的智能论文写作协作平台，主要面向需要撰写学术论文的高校学生和研究人员。系统采用前后端分离架构，支持用户注册登录、论文管理、Agent辅助写作、文献管理等功能。系统参与者分为普通用户和管理员两种角色，各角色的权限说明如表2-1所示。

| 管理员   | 系统的最高权限管理者，除普通用户功能外，还拥有用户管理、论文管理、系统日志查看等管理功能                     |

### 2.1.1 用户功能需求

**（1）注册与登录**

用户可以通过注册功能创建账号，注册时需要提供用户名和密码，系统对用户名唯一性、密码长度进行校验。登录时验证用户名和密码的正确性，验证通过后返回JWT令牌用于后续请求的身份认证。登录注册界面需提供明确的错误提示信息，如"用户名已存在""用户名或密码错误"等。

**（2）个人中心管理**

用户可以在个人中心页面查看和编辑个人信息，包括上传头像、修改用户名、修改密码等。个人中心还展示用户已创建的论文列表和管理个人文献库（即"我的收藏"），方便用户快速查找和管理自己的学术资源。

**（3）论文写作管理**

用户创建论文时需要输入论文主题、选择论文模板（课程论文模板或期刊论文模板），并可从个人文献库中勾选参考文献。创建后可查看论文列表，列表中展示论文标题、当前状态（如"草稿""文献解析中""大纲生成中""撰写中""已完成"等）和创建时间。

**（4）Agent辅助写作**

Agent辅助写作是本系统的核心功能，采用Human-in-the-loop（人在回路）的协作模式：

- **文献解析**：用户提供参考文献后，文献解析Agent自动提取每篇文献的研究问题、方法和结论，输出结构化文献摘要，用户确认或驳回后进入下一步。
- **大纲生成**：大纲生成Agent根据已确认的文献摘要和论文模板，生成包含章节标题和写作要点的论文大纲，用户可调整章节顺序、增删要点。
- **内容撰写**：内容撰写Agent按照已确认的大纲逐章撰写论文内容，支持逐章确认或全量确认。
- **润色优化**：润色Agent对论文进行语言流畅度提升、重复表达改写、术语统一等优化。
- **引用检查**：引用检查Agent校验论文中所有引用的准确性和格式规范性，输出检查报告。

用户在每个Agent产出后可以审阅、确认、驳回（附修改意见）或直接编辑修改内容，系统记录所有用户反馈用于Agent改进。

**（5）文献库管理**

用户可以在"我的文献库"页面管理个人学术文献，支持新增文献（填写标题、作者、出处、摘要等信息）、删除文献。文献库中的文献可在创建论文时被勾选引用，实现文献的复用。

### 2.1.2 管理员功能需求

**（1）用户管理**

管理员可以查看系统所有用户列表，了解用户注册情况，必要时可管理用户状态。

**（2）论文管理**

管理员可以查看系统所有论文列表，按状态筛选论文，必要时可删除违规论文。

**（3）系统日志查看**

管理员可以查看Agent执行的详细日志，包括每个Agent的输入输出、执行耗时、成功失败状态等，用于监控系统运行状况和排查问题。

**（4）系统统计**

管理员可以查看系统基础统计数据，如用户总数、论文总数等，了解系统整体使用情况。

## 2.2 非功能需求分析

### 2.2.1 性能需求

- 页面响应时间：用户操作后页面应在2秒内给出反馈。
- Agent执行时间：单个Agent执行应在30秒内完成（取决于LLM API响应速度）。
- 并发支持：系统应支持至少10个用户同时使用。

### 2.2.2 安全需求

- 用户密码使用bcrypt算法哈希存储，不存储明文密码。
- 用户认证采用JWT令牌机制，令牌有效期7天。
- 前后端通信需携带JWT令牌，后端对每个API请求进行令牌验证。
- 管理员接口需校验用户角色权限，防止越权访问。
- 注册时对用户名唯一性进行校验，防止重复注册。

### 2.2.3 可用性需求

- 界面风格统一，使用编辑主义×瑞士风格的设计语言（深蓝黑#0F172A为主色，亮蓝#2563EB为点缀色）。
- 页面布局响应式，适配不同屏幕尺寸。
- 提供清晰的错误提示信息，包括登录失败原因、Agent执行异常等。
- Agent执行过程中通过SSE实时推送状态变更，用户可随时了解进度。

### 2.2.4 可维护性需求

- 后端采用模块化架构，按功能模块（auth、user、papers、agent等）划分路由。
- Agent层封装统一基类，新增Agent只需继承BaseAgent并实现run方法。
- 所有Agent执行日志记录到数据库，便于问题追溯和系统调试。
- 代码遵循PEP 8编码规范，关键函数添加类型提示。
